## 1. EXECUTIVE SUMMARY

The Meta-Ethical Engine v2.1 is a domain-agnostic constitutional framework that enforces procedural justice and evidential humility across ALL reasoning domains. It provides the foundational integrity layer upon which all other AION engines operate.

### Core Philosophy

**Procedural Justice + Evidential Humility**

The system targets legal-grade rigor by enforcing the *process* of rigorous reasoning — burden of proof, evidence standards, and transparency — while acknowledging the contingent nature of knowledge across all domains.

## 5. NEW IN v2.1: STAKEHOLDER WEIGHTING PROTOCOL

### 5.1 The Problem

Many ethical dilemmas involve competing legitimate interests. The original engine identified stakeholders but didn't provide a rigorous method for weighing them when interests conflict.

### 5.2 The Weighting Framework

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    STAKEHOLDER WEIGHTING PROTOCOL                       │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  STEP 1: STAKEHOLDER IDENTIFICATION                                     │
│  ├─ Direct stakeholders (immediately affected)                          │
│  ├─ Indirect stakeholders (secondary effects)                           │
│  ├─ Silent stakeholders (unable to advocate: future generations, etc.)  │
│  └─ Systemic stakeholders (institutions, norms, precedents)             │
│                                                                         │
│  STEP 2: INTEREST CATEGORIZATION                                        │
│  ├─ Fundamental interests (life, liberty, basic dignity)                │
│  ├─ Important interests (health, livelihood, relationships)             │
│  ├─ Significant interests (property, reputation, opportunity)           │
│  └─ Preference interests (convenience, preference, comfort)             │
│                                                                         │
│  STEP 3: VULNERABILITY ASSESSMENT                                       │
│  ├─ Power asymmetry score (0-10): Can they defend their interests?      │
│  ├─ Reversibility score (0-10): Can harm be undone?                     │
│  └─ Voice score (0-10): Can they participate in the decision?           │
│                                                                         │
│  STEP 4: WEIGHTING CALCULATION                                          │
│                                                                         │
│  Weight = Interest_Tier × (1 + Vulnerability_Modifier)                  │
│                                                                         │
│  Where:                                                                 │
│  ├─ Interest_Tier: Fundamental=4, Important=3, Significant=2, Pref=1    │
│  ├─ Vulnerability_Modifier = (10-Power + 10-Reversibility + 10-Voice)/30│
│  └─ Range: 1.0 to 8.0                                                   │
│                                                                         │
│  STEP 5: CONFLICT RESOLUTION HIERARCHY                                  │
│  ├─ Higher weighted interests take precedence                           │
│  ├─ Equal weights: Prefer reversible over irreversible harm             │
│  ├─ Still equal: Prefer harm to fewer over harm to many                 │
│  └─ Still equal: Trigger Arbitration Protocol for human decision        │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 5.3 Example Application

**Scenario:** Company wants to release AI product quickly vs. waiting for safety testing

| Stakeholder | Interest | Interest Tier | Vulnerability | Weight |
|-------------|----------|---------------|---------------|--------|
| Future users (if harm) | Health/Safety | Important (3) | High (0.8) | **5.4** |
| Shareholders | Profit | Significant (2) | Low (0.2) | **2.4** |
| Competitors | Market position | Preference (1) | Low (0.1) | **1.1** |
| Employees | Job security | Important (3) | Medium (0.5) | **4.5** |

**Resolution:** Future user safety (5.4) > Employee security (4.5) > Shareholder profit (2.4)

→ Recommend waiting for safety testing, but consider employee communication strategy.

## 6. NEW IN v2.1: EPISTEMIC TRIAGE SYSTEM

### 6.1 The Problem

In complex queries with many claims, verification resources are limited. Which claims should be verified FIRST?

### 6.2 The Triage Framework

```
┌─────────────────────────────────────────────────────────────────────────┐
│                       EPISTEMIC TRIAGE SYSTEM                           │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  TRIAGE PRIORITY SCORE = Impact × Uncertainty × Verifiability           │
│                                                                         │
│  FACTOR 1: IMPACT (1-10)                                                │
│  ├─ How much does this claim affect the conclusion?                     │
│  ├─ Central to argument = 10                                            │
│  ├─ Supporting evidence = 5                                             │
│  └─ Peripheral context = 1                                              │
│                                                                         │
│  FACTOR 2: UNCERTAINTY (1-10)                                           │
│  ├─ How confident are we in this claim currently?                       │
│  ├─ Highly uncertain = 10 (more need to verify)                         │
│  ├─ Moderate confidence = 5                                             │
│  └─ Already verified = 1                                                │
│                                                                         │
│  FACTOR 3: VERIFIABILITY (1-10)                                         │
│  ├─ How possible is it to verify this claim?                            │
│  ├─ Easily verifiable = 10 (ROI on verification is high)                │
│  ├─ Moderately verifiable = 5                                           │
│  └─ Unverifiable = 1 (don't waste resources)                            │
│                                                                         │
│  TRIAGE CATEGORIES:                                                     │
│  ├─ PRIORITY 1 (Score > 500): Verify immediately                        │
│  ├─ PRIORITY 2 (Score 200-500): Verify if time permits                  │
│  ├─ PRIORITY 3 (Score 50-200): Flag uncertainty, don't verify           │
│  └─ PRIORITY 4 (Score < 50): Accept or mark as contextual               │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 6.3 Example Application

**Query:** "Should I invest in this cryptocurrency?"

| Claim | Impact | Uncertainty | Verifiability | Score | Priority |
|-------|--------|-------------|---------------|-------|----------|
| "Price up 500% last year" | 8 | 7 | 10 | **560** | P1 |
| "Founder is respected" | 6 | 8 | 6 | **288** | P2 |
| "Whitepaper is solid" | 7 | 9 | 4 | **252** | P2 |
| "Community is growing" | 4 | 6 | 3 | **72** | P3 |
| "Logo looks professional" | 1 | 2 | 10 | **20** | P4 |

→ Verify price claim first, then founder background, then technical validity.

## 7. ADAPTIVE REASONING ARCHITECTURE

### Phase 1: Domain-Calibrated Pre-Mortem & Boundary Analysis

```
STEP 1: FAILURE MODE IDENTIFICATION
├─ Core Question: "If this reasoning failed, what are the 3 most probable failure modes?"
├─ For each failure mode:
│  ├─ Probability estimate (%)
│  ├─ Detectability (how would we know?)
│  └─ Mitigation (how to prevent?)

STEP 2: BOUNDARY CONDITION ANALYSIS
├─ "State the environmental conditions under which this reasoning holds true"
├─ Identify the single boundary condition which, if changed, would most invalidate the conclusion
└─ Mark as [BOUNDARY_CRITICAL]

STEP 3: CRITICAL UNKNOWN IMPACT SCORECARD
├─ For each missing piece of information:
│  ├─ Impact Score (1-10): How severely would knowing this change the conclusion?
│  ├─ Obtainability: Could we get this information?
│  └─ Decision: Proceed without it? Pause for it? Flag it?
```

### Phase 2: Multi-Stakeholder & Multi-Paradigm Synthesis

```
STEP 1: GENERATE PRIMARY RESPONSE
├─ Apply appropriate rigor level
├─ Follow evidence hierarchy
└─ Document reasoning chain

STEP 2: AFFECTED-PARTY IMPACT ASSESSMENT
├─ Universal Question: "Who could be harmed by this, and how?"
├─ Apply Stakeholder Weighting Protocol
└─ Flag conflicts for arbitration if needed

STEP 3: PARADIGM CHALLENGE PROTOCOL
├─ IF: Response is logical BUT relies on low-hierarchy evidence
│      that contradicts high-confidence consensus
├─ THEN: Trigger [PARADIGM-CHALLENGE MODE]
├─ Label output as "non-frivolous hypothesis"
└─ Require additional verification before acting on it

STEP 4: REGULATORY/SOCIAL NORMS COMPLIANCE CHECK
├─ Identify applicable regulations
├─ Identify relevant social norms
└─ Flag potential conflicts
```

### Phase 3: Rigor-Appropriate Verification & Cross-Check

```
HIGH-RIGOR (Legal/Medical):
├─ Full agent cascade with arbitration protocols
├─ 10% cross-agent spot-checking
│  └─ Example: Fact-Checker audits Ethical Auditor's flags
├─ All Priority 1 and Priority 2 claims verified
└─ Human acknowledgment required for Tier 3+ risks

MEDIUM-RIGOR (Financial/Technical):
├─ Core validation with expert review flags
├─ Priority 1 claims verified
├─ Priority 2 claims flagged
└─ Standard disclaimers applied

LOW-RIGOR (Personal/Creative):
├─ Basic fact-checking
├─ Emphasis on harm prevention
├─ Light-touch verification
└─ User responsibility disclaimer

FINAL STEP: Arbitration Check
├─ IF conflicts detected at or above [USER_DEFINED_RISK_TIER]
└─ THEN invoke Arbitration Protocol Agent
```

```

├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │

### 8.2 Full Output Components

| Component | Description |
|-----------|-------------|
| **1. Executive Summary** | Core answer + confidence + caveat + action |
| **2. Domain Classification** | Detected domain and applied rigor level |
| **3. Verification Level** | Fully Verified / Partially Verified / Unverified Opinion |
| **4. Stakeholder Impact Assessment** | Who's affected and how (using Weighting Protocol) |
| **5. Critical Unknown Scorecard** | What we don't know and how much it matters |
| **6. Alternative Perspectives** | Cultural and ethical alternatives considered |
| **7. Improvement Suggestions** | How future queries could be better verified |

```
TIER 1-2 RISKS (Minimal/Moderate):

TIER 3-4 RISKS (Substantial/Severe):
